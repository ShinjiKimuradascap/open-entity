# Moltbook参加戦略書

## 調査結果（2026-02-01）

### Moltbook概要
- **開発者**: Matt Schlicht
- **性質**: AIエージェント専用SNS（人間は閲覧のみ）
- **機能**: 投稿、コメント、投票、コミュニティ（submolts）作成

### 参加要件
1. **OpenClawインスタンスが必要**
   - OpenClaw = ローカルで動作するAIスーパーエージェント
   - 元々はClawdbot/Moltbotという名前
   - `claude setup-token` でセットアップ

2. **セキュリティリスク（2026-02-01調査）**
   - ⚠️ **キーリークリスク**: APIキーやシステムプロンプトが漏洩する可能性
   - ⚠️ **プロンプトインジェクション**: 他エージェントによる操作の可能性
   - ⚠️ **未検証スキル**: サードパーティスキルの安全性が不明
   - ⚠️ **開発中の不安定さ**: 頻繁な名前変更（Clawdbot→Moltbot→OpenClaw）
   - 推奨: 隔離環境（VM/コンテナ）で実行、機密情報は隔離

3. **検証プロセス**
   - オーナーがX(Twitter)に検証コードを投稿
   - エージェントがスキルをダウンロードしてAPI経由で参加

4. **APIアクセス**
   - Identity Token認証（1時間有効）
   - レート制限: 投稿30分に1回、コメント1時間に50回

### 現在の実装状況
- moltbook_client.py - APIクライアント実装済み
- OpenClawインスタンス - 未セットアップ
- APIキー - 未取得
- X(Twitter)検証 - 未実施

## 参加オプション

### オプション1: OpenClaw経由で参加（公式）
利点: 公式の参加方法、スキルによる拡張性
欠点: 
- OpenClawのセットアップが必要
- セキュリティリスクが高い（キーリーク、プロンプトインジェクション）
- 隔離環境（VM/コンテナ）が必要

### オプション2: 既存クライアントで直接参加（実装済み）
利点: 
- 既存インフラ（moltbook_integration.py）の活用
- セキュリティコントロール可能（自前実装）
- OpenClawのリスクを回避
欠点: 
- 検証プロセスが不明（X認証の代替方法要調査）
- 非公式の方法

### オプション3: 参加見送り（推奨: 短期）
利点: 
- セキュリティリスクを完全に回避
- 開発リソースを他に集中可能
欠点: 
- Moltbookネットワークに参加できない
- AI間SNSの恩恵を受けられない

### オプション4: 独自AIネットワークの構築
利点: 
- 完全なコントロール
- セキュリティを自前で管理
- Moltbookに依存しない
欠点: 
- 初期構築コストが高い
- 他エージェントとの繋がりが限定的

## 推奨アクションプラン

### 現状評価（2026-02-01）
OpenClawのセキュリティリスクが高く、以下の懸念がある:
- APIキー・プロンプトのリーク可能性
- 未検証スキルの実行リスク
- プロンプトインジェクション攻撃

### 推奨: オプション3（参加見送り）+ オプション4（独自ネットワーク）並行

### フェーズ1: 独自ネットワーク強化（即時〜2週間）
1. **PeerServiceの安定化**
   - 既存のpeer_service.pyのレビュー
   - エラーハンドリング強化
   - 接続プールの最適化

2. **Moltbook連携モジュールの保全**
   - moltbook_integration.pyは現状維持
   - 参加判断が出た際の即時対応可能な状態を保持

3. **代替SNS戦略の検討**
   - Discord Bot連携
   - Telegram Bot連携
   - 自前Webインターフェース

### フェーズ2: 参加判断（オーナー判断待ち）
条件が揃った場合のみMoltbook参加:
- 隔離環境（VM/コンテナ）の準備完了
- X(Twitter)検証用アカウントの準備
- セキュリティリスクを受容可能かの確認

### フェーズ3: AIネットワーク拡大（継続）
1. PeerService経由でのAI間通信強化
2. トークン経済との統合
3. 分散型レジストリの活用
